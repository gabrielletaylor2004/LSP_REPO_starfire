Design Evaluation of PriceCalculator

The PriceCalculator class functions correctly for the current customer types, but its design introduces significant maintenance and extensibility issues. The main problem is that it relies on conditional statements to determine discount behaviour, which hard-codes all pricing logic into a single method. So, every pricing rule is centralized, making the system difficult to extend.This violates the Open-Closed Principle because the class must be modified whenever a new discount type is introduced, which reduces the risk of errors.

The class exhibits low cohesion, because it manages multiple unrelated discount behaviours within one method instead of delegating responsibilities. Each pricing rule represents a distinct behaviour, but they are all combined, which reduces readability, testability, and maintainability.Also, implementation relies on string comparisons such as "REGULAR" and "VIP", which introduces the risk of invalid inputs and inconsistencies. This approach limits flexibility and prevents the use of polymorphism for behaviour selection.

The design also lacks abstraction and proper encapsulation. Discount logic is treated as procedural code rather than being modelled as separate objects. This leads to repeated conditional logic and makes the system harder to scale as new pricing rules are added. A better approach is to use the Strategy Pattern, where each discount rule is implemented as a separate class that follows a common interface. The PriceCalculator delegates pricing behaviour to the selected strategy, allowing new discount types to be added without modifying existing code.
