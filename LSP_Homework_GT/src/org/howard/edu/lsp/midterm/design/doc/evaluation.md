# Design Evaluation of OrderProcessor

The OrderProcessor class has several object-oriented design problems that may lead to poor object-oriented structure, maintainability or extensibility. One major issue is that it has low cohesion and violates the single responsibility principle, meaning all classes do not have one, clear responsibility. One class (OrderProcessor) is handling customer and order data, tax calculation, receipt printing, file storage, email confirmation, discount logic, and activity logging all at once. A well-designed object-oriented system should distribute these responsibilities across multiple classes instead of concentrating everything in one place.

Another major issue is poor encapsulation. The fields customerName, email, item, and price are all public. So, any other class can directly change the internal state of the object without control or validation, which increases the chance of errors and makes the class harder to protect from invalid values. Good object-oriented design encourages hiding data behind private fields and providing controlled access through methods.

The class is also tightly coupled to specific implementation details. For example, it writes directly to a file using FileWriter, prints directly to the console using System.out.println, and logs activity using the current date inside the method. Because these details are built directly into the class, it would be difficult to change the output format, save orders to a database instead of a file, or use a different notification system without modifying the class itself. This reduces flexibility and violates the idea that classes should be open for extension but closed for modification.

Additionally, the processOrder() method is doing too much work and contains several unrelated behaviours in one long method. This makes the code harder to understand, test, and debug. If a problem occurs during tax calculation, file writing, discount processing, or logging, all of those concerns are mixed together in the same method. A better design would separate these operations into smaller classes or services with focused responsibilities.

The discount logic is also poorly placed. The method prints and saves the total before applying the discount if the price is greater than 500. This creates an inconsistency because the receipt and stored value may not match the final discounted total. This indicates weak sequencing and weak responsibility assignment. Business rules such as discounts and taxes should be handled in a consistent order, ideally by a class dedicated to pricing or order calculation.

The class also lacks abstraction. It treats customer information, pricing rules, persistence, and communication as raw procedural steps instead of modeling them as meaningful objects. In object-oriented design, concepts such as Customer, Order, Receipt, OrderRepository, and EmailService could each represent part of the system. The current design misses the opportunity to model the problem domain clearly.

From the perspective of object-oriented heuristics, this class has low cohesion and high coupling. Its responsibilities are not closely related, and it depends directly on many external details. Classes should represent a small, well-defined concept, but OrderProcessor behaves like a catch-all class. This kind of design becomes fragile as the system grows because small changes in one requirement can force changes in the same overloaded class.

Ultimately, the design is weak because it combines too many responsibilities, exposes internal data, mixes business logic with input and output details, and makes extension difficult. A stronger design would separate data from behaviour, improve encapsulation, and assign each major responsibility to a dedicated class.
